# 优先级判断规则

## 高优先级

**触发条件**（满足任一即判定为高优先级）：
1. 阻塞其他任务的关键依赖
2. 用户明确要求紧急处理
3. 安全相关问题
4. 生产环境故障
5. 数据丢失风险

**示例**:
- "修复登录漏洞"
- " urgently implement this feature"
- 数据库连接失败
- API 性能问题导致超时

---

## 中优先级

**触发条件**：
1. 常规功能开发
2. 按计划执行的迭代任务
3. 依赖其他任务但不阻塞性能问题
4. 用户体验优化

**示例**:
- 实现用户设置页面
- 添加数据导出功能
- 优化页面加载速度
- 添加单元测试

---

## 低优先级

**触发条件**：
1. 代码重构
2. 文档补充
3. 性能优化（非瓶颈）
4. 可延后的需求
5. 错误修复（不影响功能）

**示例**:
- 重构组件结构
- 补充 API 文档
- 添加日志
- 代码格式化

---

## 优先级调整

**升级场景**：
- 低优先级任务阻塞了高优先级任务 → 升级
- 用户重新强调重要性 → 升级
- 发现安全隐患 → 升至最高

**降级场景**：
- 高优先级任务依赖推迟 → 可降级
- 发现替代方案 → 可降级
